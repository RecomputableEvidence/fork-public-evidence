# Claims and Non-Claims

## Package

Fork Pilot Discovery Packet v0.1

## Supported claims

This package supports the following claims:

1. Fork has a bounded process for evaluating pilot suitability.
2. Fork pilot discovery begins with workflow selection and evidence-boundary mapping.
3. Fork distinguishes captured evidence, hashed references, external pointers, unavailable sources, and explicit non-claims.
4. Fork discovery requires stakeholder, source-system, artifact, security, and data-handling review.
5. Fork can help identify whether a workflow is appropriate for a bounded proof-of-value pilot.
6. Fork pilot discovery is separate from pilot implementation.

## Non-claims

This package does not claim:

1. Fork is ready to deploy in the prospect's environment.
2. The prospect's workflow is suitable for Fork.
3. Fork can ingest every relevant system.
4. Fork guarantees source completeness.
5. Fork proves AI output correctness.
6. Fork proves decision correctness.
7. Fork asserts legal admissibility.
8. Fork satisfies compliance obligations.
9. Fork performs audit.
10. Fork performs legal review.
11. Fork owns remediation or reporting.
12. Fork blocks, routes, approves, denies, or modifies live workflows.
13. Fork can reconstruct hidden vendor behavior.
14. Fork can recover evidence that was never observed, exported, attested, captured, or referenced.

## Boundary statement

Pilot discovery is not deployment.

Pilot discovery is the process of determining whether a bounded, read-only evidence-preservation pilot can be responsibly scoped.

Fork remains the evidence hand-off.

It does not become the upstream authority.

It does not become the downstream judgment.

# Acceptance Criteria

## Purpose

This document defines what successful pilot discovery should produce.

## Discovery is successful when

Pilot discovery is successful when the parties can answer:

1. Which workflow is under consideration?
2. Why is the workflow consequential?
3. Where does AI assistance enter the workflow?
4. What source systems may contain observable evidence?
5. Which artifacts can be captured?
6. Which artifacts can be hashed or referenced?
7. Which sources are unavailable?
8. Which claims must be explicitly excluded?
9. Which client functions own legal, compliance, audit, risk, security, remediation, and reporting responses?
10. Whether the workflow is suitable for pilot scoping.

## Required discovery outputs

A successful discovery process should produce:

- Candidate workflow description
- Stakeholder map
- Source-system inventory
- Evidence boundary worksheet
- Preliminary artifact map
- Preliminary non-claims
- Data sensitivity notes
- Security review questions
- Pilot suitability decision

## Discovery result states

Use one of the following:

### SUITABLE_FOR_PILOT_SCOPING

The workflow appears suitable for a bounded pilot scope.

### NARROWING_REQUIRED

The workflow may be suitable, but scope must be narrowed.

### SOURCE_REVIEW_REQUIRED

Additional source-system review is required.

### SECURITY_REVIEW_REQUIRED

Additional security or data-handling review is required.

### NOT_SUITABLE_AT_THIS_STAGE

The workflow is not suitable for Fork at this stage.

## Non-acceptance

Discovery is not successful if:

- The workflow cannot be described.
- The AI-assisted surface cannot be identified.
- Source systems cannot be identified.
- The client expects Fork to enforce, block, approve, or decide.
- The evidence boundary cannot be stated.
- Non-claims cannot be accepted.

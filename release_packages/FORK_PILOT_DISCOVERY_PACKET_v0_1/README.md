# Fork Pilot Discovery Packet v0.1

## Purpose

This package provides the pilot discovery orientation packet for Fork.

It is intended for serious prospects and qualified reviewers who need to determine whether a specific AI-assisted workflow is suitable for bounded, read-only evidence preservation.

This package is not a pilot-ready implementation package.

This package is not a client-specific evidence boundary packet.

This package is not a production deployment claim.

## Package role

The Pilot Discovery Packet helps structure the first serious evaluation of a possible Fork pilot.

It identifies the questions, artifacts, source systems, stakeholders, and boundaries that must be understood before any proof-of-value or design-partner pilot is scoped.

Fork should not enter a pilot conversation by claiming that all relevant evidence can be captured.

Fork should enter by asking what can be observed, what can be captured, what can only be referenced, what may be unavailable, and what must be treated as an explicit non-claim.

## Intended recipients

- Serious enterprise prospect
- Pilot sponsor
- Legal operations lead
- Compliance lead
- Audit lead
- Risk lead
- Security or GRC reviewer
- Enterprise AI governance lead
- Technical integration counterpart
- Workflow owner

## Supported claim

This package supports the claim that Fork can help evaluate whether a defined client workflow is suitable for bounded, read-only evidence preservation.

It supports pilot qualification.

It does not support a claim that the workflow is already pilot-ready.

## Use guidance

Send this package when a serious prospect asks:

"How would we evaluate a pilot?"

"What would you need from us?"

"Which workflow would make sense?"

"What systems would Fork need to observe?"

"What would Fork capture and not capture?"

Do not send this package as a substitute for a pilot-ready implementation packet.

A pilot discovery packet determines whether pilot scoping is responsible.

It does not itself authorize deployment.

# Claims and Non-Claims

## Package

Fork Client Evidence Boundary Packet Template v0.1

## Supported claim

This package supports one bounded claim:

Fork can translate a completed client discovery return packet into a client-specific evidence boundary for review before any sidecar bridge is specified.

## Supporting facts

This package provides boundary sections for:

- client workflow boundary
- source-system boundary
- evidence artifact boundary
- state-transition boundary
- access and export boundary
- security and data-handling boundary
- institutional ownership boundary
- acceptance criteria
- implementation blockers
- sidecar bridge specification placeholder

## Non-claims

This package does not claim:

1. Fork is deployed.
2. Fork is approved by the client.
3. Fork has access to client systems.
4. Fork is ready for production integration.
5. Fork can capture all relevant evidence.
6. Fork proves source completeness.
7. Fork proves AI output correctness.
8. Fork proves decision correctness.
9. Fork asserts legal admissibility.
10. Fork satisfies compliance obligations.
11. Fork performs audit.
12. Fork performs legal review.
13. Fork owns risk acceptance.
14. Fork owns remediation.
15. Fork owns reporting.
16. Fork controls workflow execution.
17. Fork blocks, routes, approves, denies, or modifies live workflow activity.
18. Fork reconstructs hidden vendor behavior unless such evidence is captured or provided within the declared boundary.
19. Fork recovers evidence that was never observed, exported, attested, captured, hashed, or referenced.
20. Fork can specify a sidecar bridge without a mapped evidence boundary.

## Boundary statement

This packet defines what may be preserved, referenced, excluded, or treated as unavailable within a specific client workflow boundary.

It does not authorize deployment.

It does not authorize system access.

It does not create institutional authority.

Fork preserves the record.

The institution owns the action.

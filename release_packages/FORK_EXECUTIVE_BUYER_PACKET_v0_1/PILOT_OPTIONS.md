# Pilot Options

## Purpose

This document provides executive-level pilot path orientation.

It does not define commercial terms.

It does not replace a pilot-ready implementation packet.

## Pilot path

A responsible pilot path should proceed in stages:

1. Public doctrine orientation
2. Executive buyer review
3. Technical validation review
4. Pilot discovery
5. Pilot-ready implementation scoping
6. Client-specific evidence boundary definition

## Option 1: Executive discovery conversation

### Purpose

Determine whether the buyer problem exists and whether the organization has candidate workflows worth evaluating.

### Output

- initial workflow candidates
- stakeholder map
- buyer concern map
- decision on whether pilot discovery is appropriate

### Non-claim

This is not technical validation or pilot scoping.

## Option 2: Technical validation review

### Purpose

Allow technical reviewers to inspect the public technical posture before pilot discovery.

### Output

- technical review notes
- questions about schemas, examples, tests, tools, package verification, and evidence states
- decision on whether pilot discovery is appropriate

### Non-claim

This is not a client deployment.

## Option 3: Pilot discovery

### Purpose

Evaluate whether a defined workflow is suitable for bounded, read-only evidence preservation.

### Output

- workflow selection
- source-system inventory
- evidence boundary worksheet
- security and data-handling questions
- pilot suitability decision

### Non-claim

This is not pilot implementation.

## Option 4: Pilot-ready implementation scoping

### Purpose

Define a bounded proof-of-value pilot after discovery.

### Output

- pilot scope
- source-system matrix
- evidence artifact map
- roles and responsibilities
- acceptance criteria
- known limitations
- client-specific non-claims

### Non-claim

This does not imply general enterprise production readiness.

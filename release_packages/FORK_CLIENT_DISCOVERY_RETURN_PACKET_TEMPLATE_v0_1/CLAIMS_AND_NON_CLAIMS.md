# Claims and Non-Claims

## Package

Fork Client Discovery Return Packet Template v0.1

## Purpose

This file states the canonical claims and non-claims for the Client Discovery Return Packet Template.

The packet is a client-completable discovery return template.

It is not an onboarding packet.

It is not a deployment package.

It is not a production integration specification.

It is not a client-specific evidence boundary.

It is not a sidecar bridge specification.

## Supported claim

This package supports one bounded claim:

Fork can receive structured client workflow information and evaluate whether a client-specific evidence boundary and sidecar bridge specification can be responsibly scoped.

## Supporting facts

This package provides structured fields for:

- client workflow profile
- source-system inventory
- access and export model
- AI-assisted surface
- evidence artifact mapping
- workflow state transitions
- security and data-handling constraints
- institutional ownership
- claims and non-claims acknowledgment
- co-integration boundary
- configuration output targets

## Non-claims

This package does not claim:

1. Fork is deployed.
2. Fork is approved for the client.
3. Fork has access to client systems.
4. Fork is ready for client onboarding.
5. Fork is ready for production integration.
6. Fork can capture all relevant evidence.
7. Fork proves source completeness.
8. Fork proves AI output correctness.
9. Fork proves decision correctness.
10. Fork asserts legal admissibility.
11. Fork satisfies compliance obligations.
12. Fork performs audit.
13. Fork performs legal review.
14. Fork owns risk acceptance.
15. Fork owns remediation.
16. Fork owns reporting.
17. Fork controls workflow execution.
18. Fork blocks, routes, approves, denies, or modifies live workflow activity.
19. Fork reconstructs hidden vendor behavior unless such evidence is captured or provided within the declared boundary.
20. Fork recovers evidence that was never observed, exported, attested, captured, hashed, or referenced.

## Boundary statement

Completion of this packet is not deployment approval.

Completion of this packet is not implementation authorization.

Completion of this packet means only that Fork has received structured information that may allow review of whether a client-specific evidence boundary and sidecar bridge can be responsibly scoped.

## Relationship to acknowledgment file

`CLAIMS_AND_NON_CLAIMS_ACKNOWLEDGMENT.md` is the client-facing acknowledgment worksheet.

This file is the canonical package-level claims and non-claims statement.

Both should remain in the package.

## Governing principle

The client discovery return packet maps the environment.

The client evidence boundary defines the pilot.

The sidecar bridge follows the boundary.

Fork preserves the record.

The institution owns the action.

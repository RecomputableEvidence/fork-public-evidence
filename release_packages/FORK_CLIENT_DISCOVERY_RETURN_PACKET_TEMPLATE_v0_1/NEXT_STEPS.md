# Next Steps

## For the client or qualified prospect

Complete the packet files with accurate information.

Unknowns should remain marked as `UNKNOWN`.

Do not guess.

Do not remove unresolved constraints.

Return the completed packet as a folder or compressed archive.

## For Fork

After receiving a completed packet, review for:

1. workflow specificity
2. source-system clarity
3. access/export feasibility
4. evidence artifact classification
5. AI-assisted surface clarity
6. state-transition clarity
7. security and data-handling constraints
8. institutional ownership
9. accepted non-claims
10. co-integration boundaries, if applicable
11. configuration output targets
12. implementation blockers

## Possible review outcomes

Fork may classify the returned packet as:

- `DISCOVERY_RETURN_REVIEWABLE`
- `DISCOVERY_RETURN_INCOMPLETE`
- `DISCOVERY_RETURN_BLOCKED`
- `CLIENT_EVIDENCE_BOUNDARY_READY_TO_DRAFT`
- `NOT_SUITABLE_FOR_FORK_AT_THIS_STAGE`

## Possible next artifacts

If the returned packet is reviewable, Fork may prepare:

- client evidence boundary packet
- sidecar bridge specification
- implementation blockers report
- pilot-ready implementation packet
- technical validation addendum
- co-integration boundary addendum

## Boundary close

A returned discovery packet is not deployment approval.

It is the basis for determining whether a client-specific evidence boundary and sidecar bridge can be responsibly scoped.

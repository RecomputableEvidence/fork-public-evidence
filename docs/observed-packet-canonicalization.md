# Observed Evidence Packet canonicalization and file layout

This document defines the educational packet layout used by `examples/observed-evidence-packets/`. It is a teaching artifact, not a replacement for Fork's main verification-result contract.

## Packet root layout

```text
evidence-packet-<id>/
  manifest.json                # canonical packet facts only
  artifacts/
    brief.txt
    generation.log
  inputs/                      # optional input snapshots for recomputation examples
  generator/                   # optional deterministic generator for STRONG_RECOMPUTATION examples
  SHA256SUMS                   # authoritative digest surface
  receipt.json                 # optional real detached receipt, never a template placeholder
  verification_result.json     # verifier output, generated by CI
  README.md                    # optional packet-local explanation
```

## Manifest boundary

`manifest.json` is the canonical fact surface. It must not contain:

- `manifest_digest`
- `signatures`
- `gate_receipts`

The manifest contains artifact digests and byte sizes as packet facts. The verifier confirms these match `SHA256SUMS` and actual file bytes.

## Digest rules

Canonical manifest bytes are produced as:

```python
json.dumps(manifest_obj, sort_keys=True, separators=(",", ":")).encode("utf-8")
```

`SHA256SUMS` stores:

```text
sha256:<manifest_digest>  manifest.json
sha256:<artifact_digest>  artifacts/<name>
```

For this educational suite, `SHA256SUMS` is authoritative and is checked against both manifest-declared artifact digests and actual file hashes.

## Receipt and signature rules

Real `receipt.json` files, if present, must be detached from the manifest and must reference the canonical manifest digest as `sha256:<manifest_digest>`.

This educational suite does not implement cryptographic signature verification. Signature verification is explicitly reported as `NOT_CHECKED`. A future production version may add GPG or OpenSSL verification using public keys under `keys/`.

## Recomputability class rules

The executable examples only use:

- `NON_RECOMPUTABLE`
- `STRONG_RECOMPUTATION`

Experimental classes such as `PARTIAL_RECOMPUTATION` or `PRESERVED_ONLY` are not schema-valid in this suite and must not be introduced as executable classes without a separate doctrine update.

# Fork bounded starter tasks v0.1

Status: `DRAFT_STARTER_TASK_REGISTER`

## Purpose and standing

This register gives a newcomer three concrete ways to contribute without searching an undifferentiated repository or inventing a task. Each task is deliberately bounded by files, evidence, stop conditions, and non-effects.

Selecting or completing a task does not admit a contribution, validate Fork, establish reviewer competence, authorize execution, or create publication, endorsement, compliance, legal, security, production-readiness, or institutional standing.

Before beginning, record the exact base commit and read:

- `docs/review/PUBLIC_REVIEW_QUICKSTART_v0_1.md`;
- `docs/review/FORK_REPOSITORY_REVIEW_POSTURE_v0_1.md`;
- `docs/coherence/front-door-routing-v0.1/STATUS_GLOSSARY_v0_1.md`.

## Choose by skill and available environment

| Task | Best fit | Expected effort | Required environment | Primary output |
|---|---|---:|---|---|
| `STARTER-ROUTE-001` | Technical writer, information architect, newcomer reviewer | 30–90 minutes | Browser; optional Git checkout | Exact-head route observation or bounded editorial correction |
| `STARTER-RECOMPUTE-001` | Engineer, auditor, reproducibility reviewer | 45–120 minutes | Git, Python, and PowerShell or documented fallback | Exact-head recomputation return |
| `STARTER-ADVERSARIAL-001` | Test engineer, security-minded contributor, researcher | 1–3 hours | Git and Python | One bounded adversarial-case proposal with expected rejection basis |

## STARTER-ROUTE-001 — Inspect one visitor route

### Objective

Starting from the root README, follow exactly one role route—reviewer, contributor, governance/buyer reader, or researcher—and identify a broken link, circular route, hidden prerequisite, ambiguous term, or material overread risk.

A report finding no defect is valid when the inspected route, exact head, and limitations are recorded.

### Prerequisites

- Ability to browse Markdown and links.
- No prior Fork expertise required.
- Record whether navigation coaching was provided.

### Files and surfaces in scope

- `README.md`;
- `docs/coherence/front-door-routing-v0.1/ROLE_PATH_OUTLINES_v0_1.md`;
- the first-click destination and direct supporting links on the selected route;
- `docs/coherence/front-door-routing-v0.1/NEWCOMER_WALKTHROUGH_FORMS_v0_1.md` as the observation instrument.

### Protected exclusions

Do not change:

- checker, schema, fixture, registry, experiment, receipt, or workflow semantics;
- historical, frozen, admitted, or canonical records;
- PR #84 or PR #86 artifacts or standing;
- Pair-001 state, retry authority, provider configuration, or readiness;
- repository settings or `main`.

A discovered normative or executable defect must be reported and separated into its own appropriately classified task or PR.

### Evidence to return

- exact commit or PR head;
- selected role route;
- links followed in order;
- observed wording or failure;
- `PASS`, `FAIL`, `UNRESOLVED`, or `NOT_CHECKED` only where the scope is named;
- what was not inspected;
- proposed minimal correction, if any;
- explicit statement of what the observation does not establish.

### Review path

Use the applicable section of `NEWCOMER_WALKTHROUGH_FORMS_v0_1.md`. For a file change, open one draft `E0 — Editorial` PR containing only the bounded routing correction and the preserved observation.

### Effects

- Admission effect: `NONE`.
- Provider-call effect: `NONE`.
- Experiment-execution effect: `NONE`.
- Readiness effect: `NONE`.

## STARTER-RECOMPUTE-001 — Return one exact-head public recomputation

### Objective

Run Fork's named public verifier at an exact checkout, or use the documented platform fallback when the named PowerShell verifier cannot execute. Return the observed outputs without converting a pass into truth, approval, compliance, completeness, or authority.

### Prerequisites

- Git and Python.
- Windows PowerShell or PowerShell Core for the named verifier; otherwise use and explicitly classify the documented fallback.
- Ability to preserve raw command output or a faithful transcript.

### Files and surfaces in scope

- `scripts/verify_public_review_package_v0_1.ps1`;
- `docs/review/PUBLIC_REVIEW_QUICKSTART_v0_1.md`;
- `docs/review/PUBLIC_VERIFIER_PLATFORM_FALLBACK_v0_1.md`;
- output generated by the documented commands in the reviewer's own environment.

No repository edit is required for a successful return.

### Protected exclusions

Do not:

- modify commands merely to obtain a favorable result without reporting the change;
- report fallback reconstruction as execution of the named PowerShell verifier;
- omit failures, missing dependencies, access limits, or platform barriers;
- infer repository-wide completeness from the declared checker scope;
- perform provider calls or Pair-001 execution.

### Evidence to return

- exact commit SHA;
- operating system, shell, Git, Python, and PowerShell availability/version;
- exact command executed;
- exit code and complete high-level result;
- raw output or preserved output artifact;
- whether the named verifier or fallback was used;
- failed, unresolved, and not-checked properties;
- explicit non-claims.

### Review path

Return an issue, PR review observation, or review packet bound to the exact head. A code or documentation PR is unnecessary unless a reproducible defect is found. Any corrective PR must remain separate from the recomputation return.

### Effects

- Admission effect: `NONE`.
- Provider-call effect: `NONE`.
- Experiment-execution effect: `NONE`.
- Readiness effect: `NONE`.

## STARTER-ADVERSARIAL-001 — Propose one bounded negative case

### Objective

Identify one concrete way a reader or structural checker could accept a representation that should remain rejected, unresolved, or not checked. Produce one minimal adversarial case against an existing non-preregistered public review surface.

Preferred starting surface:

- `docs/review/boundary-pressure/`;
- `tools/check_boundary_pressure_review_cases_v0_1.py`;
- the associated focused tests and fixture family already referenced by the public README.

### Prerequisites

- Ability to read JSON or Markdown fixtures and run Python tests.
- State the expected invalid condition before changing code.
- Begin from an exact base and reproduce current behavior first.

### Files and surfaces in scope

- one new or minimally changed adversarial fixture under the selected existing fixture family;
- one focused test or expected-outcome record needed to demonstrate the case;
- a short finding record preserving the pre-correction behavior when a defect is reproduced.

If checker behavior must change, separate the fixture/finding from the implementation correction when they are different change classes.

### Protected exclusions

Do not:

- modify a preregistered experiment, frozen corpus, raw output, or historical specimen;
- erase or rewrite the original failing representation;
- introduce provider calls, retries, or Pair-001 execution;
- assign admission, publication, certification, security, compliance, or readiness standing;
- broaden a localized failure into a repository-wide conclusion;
- combine unrelated editorial, checker, historical, and admission changes.

### Evidence to return

- exact base and candidate head;
- adversarial hypothesis and expected observation stated before correction;
- minimal input or fixture;
- command, exit code, and raw result;
- whether the result is `PASS`, `FAIL`, `UNRESOLVED`, or `NOT_CHECKED` within the named test scope;
- preserved negative evidence;
- affected and unaffected claims;
- residuals and explicit non-claims.

### Review path

Open one draft PR with the correct change classification. Request review of the exact head. A green workflow establishes only the declared structural checks and does not authorize merge or admission.

### Effects

- Admission effect: `NONE` unless a later, separately governed act occurs.
- Provider-call effect: `NONE`.
- Experiment-execution effect: `NONE`.
- Readiness effect: `NONE`.

## Stop and ask for classification when

Stop rather than infer when:

- the exact base cannot be established;
- the task reaches a frozen, admitted, historical, or preregistered surface;
- private evidence or third-party authorization would be required;
- an editorial observation becomes a normative contract or executable change;
- the expected result depends on provider access or execution authority;
- a failure cannot be reproduced;
- the proposed correction would erase the original failure.

## Completion rule

A starter task is complete only when its evidence return names the exact head, inspected or executed scope, observed result, failures and unresolved conditions, not-checked properties, and non-claims. Completion does not imply merge, admission, publication, endorsement, or general validity.

# Fork Client Intake Readiness Statement v0.1

## Status

Client-intake readiness statement.

This document defines what Fork is ready to do now, what Fork is not claiming to do, and how qualified client or partner conversations should be framed.

## Purpose

Fork has reached a controlled client-intake stage.

That does not mean Fork is claiming generic production deployment readiness.

It means Fork now has enough public doctrine, release-package structure, technical validation orientation, discovery scaffolding, return-packet structure, checker tooling, and synthetic example coverage to support qualified client discovery and evidence-boundary mapping.

The purpose of this statement is to prevent restraint from being misread as immaturity.

Fork current posture is deliberate:

- ready for qualified client intake;
- ready for structured discovery-return review;
- ready for client-specific evidence-boundary mapping;
- not ready to claim generic production onboarding;
- not ready to assume a universal sidecar bridge;
- not positioned as runtime control, audit automation, legal judgment, or compliance ownership.

## Readiness statement

Fork is ready for qualified client discovery and evidence-boundary mapping.

Fork is not represented as generic production onboarding.

A client-native sidecar bridge is specified only after a candidate workflow, source systems, access/export model, evidence artifacts, security constraints, institutional ownership, and accepted non-claims are mapped.

## What Fork is ready for now

Fork is ready to support the following intake-stage activities:

1. Qualified discovery conversations with potential clients, partners, or implementation-adjacent reviewers.
2. Use of the Fork Pilot Discovery Packet to determine whether a candidate AI-assisted workflow is worth mapping.
3. Client completion of a Client Discovery Return Packet.
4. Structural review of a completed Client Discovery Return Packet using the client discovery return checker.
5. Classification of returned discovery material as REVIEWABLE, INCOMPLETE, or BLOCKED.
6. Drafting of a client-specific evidence boundary when a returned discovery packet is structurally reviewable.
7. Identification of candidate sidecar bridge requirements after the evidence boundary is understood.
8. Separation of captured evidence, hashed references, external pointers, unavailable sources, out-of-scope materials, explicit non-claims, and institutional ownership.
9. Preservation of the distinction between evidence integrity and institutional action.

## Operating distinction

The next step is not installation.

The next step is workflow mapping.

Qualified intake begins with a candidate workflow.

A client discovery return packet maps workflow purpose, consequence-bearing action, source systems, access and export model, AI-assisted surface, evidence artifacts, state transitions, security and data-handling constraints, institutional ownership, accepted non-claims, co-integration boundaries, and configuration output targets.

Only after that mapping can Fork responsibly evaluate whether a client-specific evidence boundary and sidecar bridge specification can be drafted.

## Intake proof chain

Fork now supports the following intake proof chain:

Blank client return template -> INCOMPLETE

Synthetic completed client return -> REVIEWABLE

Future real client return -> REVIEWABLE / INCOMPLETE / BLOCKED

This distinction matters because it prevents Fork from treating all inbound interest as deployment readiness.

A blank packet is not usable.

A structurally completed packet may be reviewable.

A packet with access, security, ownership, or non-claim blockers may be blocked.

## Checker classifications

### REVIEWABLE

REVIEWABLE means the returned discovery packet contains enough structured information for Fork to evaluate whether a client-specific evidence boundary can be drafted.

It does not mean the client is suitable, the workflow is suitable, deployment is approved, access is granted, legal review is complete, compliance obligations are satisfied, or a sidecar bridge has been implemented.

### INCOMPLETE

INCOMPLETE means the returned discovery packet does not yet contain enough structured information for responsible review.

Common causes include unresolved placeholders, unknown workflow identification, missing source-system details, missing access/export model, unchecked non-claim acknowledgments, missing AI-surface information, missing evidence sufficiency selection, or incomplete ownership mapping.

### BLOCKED

BLOCKED means the returned discovery packet includes information that prevents responsible review or implementation-path scoping unless resolved.

Common causes include no access available, access model blocked, security blockers, implementation blockers, unaccepted non-claims, unresolved ownership conditions, or restrictions that prevent a bounded evidence boundary from being defined.

## Partner posture

Fork should not be retrofitted into another party existing deployment promises as an afterthought.

For partners, the correct posture is:

Fork is available as a client-intake-ready evidence-preservation layer for future deployments where independent recomputable evidence needs to be scoped explicitly.

A partner integration should begin only when the client environment is concrete, the workflow is identified, the source systems are known, the execution-governance layer and evidence-preservation layer can remain distinct, Fork non-claims are accepted, and the evidence boundary can be drafted without inheriting partner authority.

## Client posture

For direct client conversations, the correct posture is:

Fork is ready for qualified discovery.

The next step is not installation.

The next step is completing a discovery return packet for one candidate workflow.

If that return is reviewable, Fork can draft a client-specific evidence boundary and candidate sidecar bridge requirements.

## Boundary of the sidecar bridge

Fork does not assume a generic sidecar bridge.

The sidecar bridge is discovered from the client environment.

Candidate bridge patterns may include manual export, file drop, watch folder, read-only API pull, hybrid reference, hash-only reference, and external pointer handling.

The final sidecar bridge specification follows the accepted evidence boundary.

It does not precede it.

## Why this posture matters

The first institutional reflex after Fork preserves evidence of a failure, gap, stale context, unavailable source, or non-checked state may be to ask why Fork did not prevent the issue.

That question reframes evidence as control.

Fork boundary must hold.

Fork preserves the evidence.

The institution owns action, risk acceptance, remediation, reporting, legal interpretation, compliance posture, and workflow control.

# Public Surface Cleanup and Computed Scenario 09 Plan v0.1

## Status

Draft implementation plan for Fork AHI public proof surface hardening.

## Purpose

This plan turns reviewer feedback into a bounded cleanup and maturity step.

The reviewer validated the Scenario 08/09 split as a real governance and security failure class. The concern is narrower: the public review path still has reviewer-facing cleanup issues, and current proof language can be read more broadly than the current checkers support.

This plan separates two tracks:

1. Public surface cleanup before broader cold review.
2. Computed Scenario 09 v0.2 design, where the split-state visibility gap is derived from independent System A/B/C inputs rather than authored directly into a fixture.

## Non-claims

This plan does not assert that Fork approves, certifies, authorizes, determines compliance, determines admissibility, establishes legal sufficiency, decides acceptance, judges correctness, determines negligence, determines excuse, or determines execution eligibility.

This plan does not assert that the current AHI proof surface performs behavioral detection from independent live systems.

This plan does not replace checker scripts, scenario artifacts, reviewer packets, or local verification instructions.

## Reviewer feedback being addressed

The review produced four useful conclusions:

1. Scenario 08 and Scenario 09 are the correct split.
2. The failure class maps to real security, compliance, licensing, credentialing, access-control, and KYC/AML workflows.
3. The current AHI surface demonstrates bounded fixture consistency, internal alignment, non-claim discipline, and deterministic viewer reconstruction.
4. The current AHI surface should not be described as independently deriving split-state detection from modeled System A/B/C behavior.

The key wording correction is:

```text
Current AHI v0.1 proof surface:
bounded fixture consistency, reconstruction, non-claim, and viewer determinism checks.

Target AHI v0.2 proof surface:
computed transition-state derivation from independent System A/B/C inputs.
```

## Track A - public surface cleanup

### A1. Encoding cleanup

Before additional cold legal, compliance, audit, or GRC review, run a repository-wide UTF-8 and mojibake cleanup pass.

Required outcomes:

- no reviewer-facing titles or headings render mojibake;
- no viewer bundle contains mojibake strings;
- no comparison labels contain mojibake strings;
- no white paper, public checklist, or spec file contains common UTF-8/cp1252 corruption;
- cleanup is verified by a repository-wide script that reads files as explicit UTF-8.

Primary script:

```text
scripts/check_no_mojibake_utf8_v0_1.ps1
```

### A2. Viewer data regeneration

After encoding cleanup, regenerate viewer data bundles and rerun deterministic viewer checks.

Expected commands:

```powershell
powershell -ExecutionPolicy Bypass -File scripts\build_ahi_viewer_data_v0_1.ps1 -ForceOverwrite
powershell -ExecutionPolicy Bypass -File scripts\check_ahi_viewer_v0_1.ps1 -CheckDeterminism
powershell -ExecutionPolicy Bypass -File scripts\check_ahi_viewer_v0_2.ps1 -CheckDeterminism
```

### A3. Reviewer packet limitation language

Add explicit limitation language that says the current checkers verify bounded fixture consistency and reconstruction posture, not independently derived behavioral detection.

Primary document:

```text
docs/reviewer/AHI_REVIEWER_PACKET_LIMITATIONS_v0_1.md
```

### A4. Public review readiness condition

Do not describe the current surface as broader cold-review-ready until:

- mojibake cleanup passes;
- viewer bundles are regenerated;
- deterministic viewer checks pass from a clean tree;
- reviewer limitation language is linked from the reviewer packet or release index;
- Scenario 08/09 language distinguishes current fixture consistency verification from future computed derivation.

## Track B - computed Scenario 09 v0.2

## Goal

Create a Scenario 09 v0.2 variant where Fork derives a revocation visibility gap from independent System A/B/C state inputs.

The current Scenario 09 records the split-state failure mode. The next variant should compute it.

## Required independent inputs

Computed Scenario 09 should use separate files for independent state:

1. System A source-of-truth revocation state.
2. System B synchronization or cache state.
3. System C reliance attempt state.
4. Freshness or tolerance policy.
5. Expected derived result generated by the checker, not authored as the primary source.

Candidate files:

```text
examples/simulations/governance-proof-surface/computed/scenario_09_v0_2/system_a_revocation_state.json
examples/simulations/governance-proof-surface/computed/scenario_09_v0_2/system_b_visibility_state.json
examples/simulations/governance-proof-surface/computed/scenario_09_v0_2/system_c_reliance_attempt.json
examples/simulations/governance-proof-surface/computed/scenario_09_v0_2/freshness_tolerance_policy.json
examples/simulations/governance-proof-surface/computed/scenario_09_v0_2/derived_gap_result.json
```

## Derived conditions

The checker should compute:

- whether System A records a validity-changing event;
- whether System B had visibility into that event at the relevant time;
- whether System C consumed the changed state before relying;
- whether the gap exceeded the declared tolerance policy;
- whether current revalidation evidence is required;
- whether the output remains non-authoritative.

## Candidate derived classifications

The computed result should distinguish at least four cases:

1. `NO_GAP`
   - System B has current visibility and System C consumed current state before reliance.

2. `GAP_WITHIN_TOLERANCE`
   - A visibility or consumption gap existed but was within declared tolerance.
   - Fork records the gap and tolerance posture without deciding acceptability.

3. `REVOCATION_VISIBILITY_GAP_RECORDED`
   - System A changed validity, but System B did not yet have visibility at the relevant time.

4. `SPLIT_STATE_CONSUMPTION_GAP_RECORDED`
   - System B had visibility, but System C did not consume or reconcile the changed state before relying.

## Non-authority rule

Even when the checker derives a gap, the checker must not produce:

- approval;
- disapproval;
- risk score;
- compliance determination;
- admissibility determination;
- legal sufficiency determination;
- fault determination;
- negligence determination;
- excuse determination;
- execution eligibility determination;
- production readiness determination.

The derived result should remain a boundary-state record.

## Scenario 10 dependency

Scenario 10 should not be started as a broad new failure mode until the model defines what closes a Scenario 09 gap.

Scenario 10 candidate title:

```text
Scenario 10 - Acknowledgment, Synchronization, and Reconciliation Boundary
```

Core invariant:

```text
notification does not imply receipt
receipt does not imply consumption
consumption does not imply reconciliation
reconciliation does not imply authorization
```

Scenario 10 should define closure evidence, not fault.

## Proposed implementation sequence

1. Add this plan and limitation document.
2. Add repository-wide no-mojibake checker.
3. Run cleanup and regenerate viewer bundles.
4. Patch reviewer packet or release index to link limitation language.
5. Design computed Scenario 09 v0.2 input schema.
6. Add computed checker that derives gap state from independent inputs.
7. Add positive and negative fixtures.
8. Integrate computed checker into the main AHI checker.
9. Update reviewer packet with a separate computed derivation section.
10. Only then proceed to Scenario 10.

## Completion criteria for this plan

This plan is complete when:

- the three reviewer documents exist;
- repository-wide mojibake checker exists;
- no-mojibake checker passes;
- reviewer-facing limitation language exists;
- the computed Scenario 09 v0.2 design is ready to implement;
- no new authority, compliance, legal sufficiency, fault, or execution claims are introduced.

# Fork Local Recomputation Procedure v0.1

Status: Repository verification procedure.  
Normative force: Maintainer-governed local validation and reporting requirements for contributions.  
Scope: Local execution used to support pull-request review in `fork-public-evidence`.  
Non-claim: A successful local result does not establish truth, correctness of underlying workflow data, compliance, legal sufficiency, safety, authorization, approval, production readiness, endorsement, or institutional authority.

## Purpose

This procedure defines how contributors select, execute, and report local verification commands without creating false equivalence among shells, environments, checkers, or results.

It does not replace component-specific instructions. The smallest applicable specification, package guide, experiment procedure, or checker documentation controls the component being changed.

## Command sources

Use command sources in this order:

1. a frozen or versioned artifact-specific verification procedure;
2. an experiment preregistration or recomputation guide;
3. a component README or checker help contract;
4. [`../VERIFICATION_COMMANDS_v0_1.md`](../VERIFICATION_COMMANDS_v0_1.md);
5. the applicable CI workflow when no more specific public command is documented.

When two sources conflict, do not silently choose one. Identify the conflict in the pull request and follow the more specific governing artifact unless it is demonstrably stale or defective. A correction to the command source may need a separate change.

## Current repository-level paths

The current repository command map documents an integrated Windows proof-surface command:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\verify_fork_proof_surface_v0_1.ps1
```

That command is documented for Windows PowerShell 5.1.

The repository also documents cross-platform Python component checks, including:

```text
python tools/check_fork_proof_surface_state_v0_1.py --json --check-summary
python tools/validate_json_schema_bundle_v0_1.py --json
python tools/check_reviewer_access_path_integrity_v0_1.py --json
python tools/check_public_review_round_006_observations_v0_1.py --json
python tools/check_cross_system_claim_handoff_v0_1.py --json
```

Do not infer a repository-wide PowerShell 7+ requirement. Tool and shell requirements are component-specific unless a separately governed toolchain policy establishes a repository-wide baseline.

## Canonical command rule

A command is canonical for a surface when the governing artifact explicitly designates it as the verification or recomputation path.

A contributor must execute the canonical command when the changed surface requires it.

A translated or alternative command may provide useful supplemental evidence, but it must not be reported as equivalent unless equivalence is supported by:

- the same identified inputs;
- the same governing rules and tool version;
- materially equivalent environment assumptions;
- the same expected findings, ordering, exit behavior, and generated bytes where byte identity is required;
- an explicit repository statement or recorded comparison establishing the supported alternative path.

## Environment reporting

For each material command, record:

- operating system;
- shell and version when shell behavior is material;
- Python, Go, Node, or other runtime version when material;
- working directory;
- repository commit SHA;
- environment variables or flags that affect results;
- exact command;
- exit code;
- test count, findings, output digest, or receipt path as applicable.

Do not omit an environment limitation that prevents another reviewer from reproducing the result.

## PowerShell, Bash, WSL, and line endings

Fork does not prohibit Bash, WSL, Linux, macOS, or PowerShell 7 as general development environments.

The restriction is narrower:

- when a component declares a PowerShell command as canonical, run that command in its supported PowerShell environment;
- when a component declares a cross-platform Python path, it may be run in any supported shell without representing shell adaptation as an architectural change;
- do not convert shell-specific quoting, here-strings, interpolation, path separators, encoding, or line endings and assume equivalence without testing;
- preserve UTF-8 text and LF line endings where required by the repository review posture;
- record CRLF/LF normalization or generated-byte differences when they affect digests, fixtures, receipts, or parsing.

A Bash or WSL failure caused by an unsupported translation does not by itself invalidate the canonical Windows path. A Windows success does not by itself establish cross-platform portability.

## Selecting the test scope

Run the smallest complete set that covers the contribution's semantic effect.

### Editorial changes

At minimum:

- inspect the rendered Markdown or affected syntax;
- confirm links and paths;
- run any applicable documentation, YAML, JSON, or package integrity check;
- state why no behavioral test is required.

### Implementation changes

At minimum:

- run the affected checker or component suite;
- run regression coverage for the repaired behavior;
- run invalid/adversarial cases where applicable;
- compare output ordering, exit codes, and generated artifacts.

### Schema or contract changes

At minimum:

- validate the schema or contract bundle;
- run conforming and nonconforming fixtures;
- run affected consumers and regression suites;
- document versioning, compatibility, and migration behavior.

### Canonical, experiment, historical, or admission changes

Follow the controlling registry, preregistration, recomputation, amendment, or publication procedure. Repository-wide tests alone are insufficient when the governing surface requires exact receipts, independent execution, immutable coordinates, or append-only admission records.

## Generated outputs

Generated evidence must be attributable to inputs and commands.

Before committing generated output:

1. start from a clean working tree or record pre-existing differences;
2. run the designated generation command;
3. inspect every changed file;
4. confirm deterministic fields, ordering, timestamps, paths, and digests;
5. rerun the verifier against the generated output;
6. record any nondeterminism or machine-specific field.

Do not manually edit a generated receipt unless its governing schema explicitly allows manual authorship.

When generation changes unrelated files or bytes, stop and investigate. Do not normalize unexpected differences merely to obtain a clean pass.

## Result interpretation

Report results with their declared scope.

Preferred examples:

- “The schema bundle validator returned exit code 0 for commit `<sha>`.”
- “The canonical fixture suite produced 18 expected conforming results and 15 expected invalid results.”
- “The checksum verifier confirmed byte agreement with the listed manifest.”

Avoid unbounded statements such as:

- “Fork is verified.”
- “The evidence is correct.”
- “The repository is secure.”
- “The workflow is compliant.”
- “The receipt proves the event.”

A pass means only that the identified command produced the identified bounded result under the identified inputs, environment, and governing rules.

## Verification record template

Use the following structure in a pull request when the result is material:

```text
Commit: <full SHA>
Change class: <E0/I1/S2/C3/X4/H5/A6>
Operating system: <name and version>
Shell: <name and version, if material>
Runtime: <name and version>
Working directory: <path relative to repository root>
Command: <exact command>
Exit code: <integer>
Observed result: <test count, finding, digest, or receipt path>
Expected result: <bounded expected result>
Limitations: <not executed, unavailable, environment differences, or none observed>
```

## Independent recomputation

When independent recomputation is required, the independent reviewer should use:

- a fresh clone or otherwise attributable repository state;
- the exact admitted or proposed commit;
- documented dependencies;
- the canonical command;
- independently captured output;
- an explicit record of inaccessible inputs, deviations, or failed steps.

The contributor's local result may be compared with the independent result, but it must not be treated as the independent evidence itself.

## Failure handling

A failed command is evidence about that execution, not automatically evidence of a defect in the underlying artifact.

Preserve and classify:

- command or setup errors;
- missing dependencies;
- unsupported environment behavior;
- deterministic checker failures;
- fixture expectation mismatches;
- inaccessible or missing source evidence;
- nondeterministic output;
- unresolved causes.

Do not convert “not executed,” “not accessible,” or “unsupported environment” into “pass,” “absent,” or “invalid.”

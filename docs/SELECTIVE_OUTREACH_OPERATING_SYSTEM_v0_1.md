# Selective Outreach Operating System v0.1

## Purpose

This document defines Fork's selective outreach operating system.

It exists to prevent Fork outreach from collapsing into a generic AI governance pitch.

Fork now has three external paths:

1. Public reviewer path
2. Advisor / design-partner path
3. Enterprise bounded workflow PoV path

Each path has a different audience, asset anchor, tone, and success condition.

## Boundary

This document is an outreach planning aid.

It is not:

- a sales script;
- a pricing commitment;
- a procurement representation;
- a customer deployment claim;
- a production-readiness claim;
- a compliance certification;
- a legal opinion;
- a security approval;
- a public endorsement record.

Do not commit populated target lists, private relationship notes, warm-introduction details, or prospect research to the public repository.

Use public-safe templates for structure only.

## Operating principle

Fork outreach should optimize for serious signal, not volume.

The first objective is not mass awareness.

The first objective is to create two serious conversations with people or organizations that understand the need for recomputable evidence in AI-assisted workflows.

## Three external paths

### 1. Public reviewer path

Audience:

- open-source reviewers;
- AI governance researchers;
- technical reviewers;
- evidence, assurance, or auditability researchers;
- standards-adjacent technologists.

Goal:

- critique;
- validation;
- visibility;
- adversarial review;
- vocabulary pressure-testing.

Asset anchors:

- `README.md`
- `REVIEWER_QUICK_START_v0_1.md`
- `docs/REVIEWER_START_HERE_v0_1.md`
- `docs/VERIFICATION_COMMANDS_v0_1.md`
- white paper.

Tone:

- transparent;
- inspectable;
- bounded;
- intellectually honest.

Do not ask this audience to buy.

### 2. Advisor / design-partner path

Audience:

- senior operators;
- legal-tech builders;
- AI governance leads;
- model-risk practitioners;
- GRC or audit innovation leaders;
- people who can pressure-test enterprise fit.

Goal:

- stress-test;
- sharpen buyer language;
- identify edge cases;
- create signal;
- find warm introductions.

Asset anchors:

- `docs/ENTERPRISE_DISCOVERY_POV_PACKET_v0_1.md`
- `docs/BOUNDED_WORKFLOW_POV_SCOPE_TEMPLATE_v0_1.md`
- `docs/WORKED_EXAMPLE_VENDOR_RISK_AI_ASSISTED_DECISION_RECORD_v0_1.md`
- `docs/FORK_NON_CLAIM_BOUNDARY_v0_1.md`.

Tone:

- help us test this before enterprise rollout;
- narrow;
- non-performative;
- serious.

Do not present this as a broad advisory-board ask unless that is explicitly intended.

### 3. Enterprise bounded workflow PoV path

Audience:

- GC;
- CCO;
- Head of AI Risk;
- Head of Model Risk;
- Audit innovation lead;
- CTO / CISO-adjacent sponsor;
- AI governance program owner.

Goal:

- 30-minute fit conversation;
- one bounded workflow identified;
- paid proof-of-value conversation;
- enterprise discovery path.

Asset anchors:

- `docs/ENTERPRISE_DISCOVERY_POV_PACKET_v0_1.md`
- `release_packages/FORK_EXECUTIVE_BUYER_PACKET_v0_1/ENTERPRISE_DISCOVERY_POV_v0_1.md`
- `release_packages/FORK_EXECUTIVE_BUYER_PACKET_v0_1/NEXT_STEPS.md`
- `docs/BOUNDED_WORKFLOW_POV_SCOPE_TEMPLATE_v0_1.md`.

Tone:

- bounded risk reduction;
- evidentiary reconstruction;
- narrow PoV;
- no platform rollout promise.

Do not lead with the full repository.

## Targeting rule

A target should not be added merely because the organization is large or famous.

A target should have at least one of:

- active GenAI deployment pressure;
- public AI governance posture;
- regulated workflow exposure;
- audit, risk, legal, or compliance reconstruction pain;
- prior publication on AI governance, auditability, evidence, or model risk;
- likely internal champion;
- warm-introduction path;
- existing relationship.

## First-wave target count

Start with 12 total:

```text
4 public reviewers
4 advisor / design-partner candidates
4 enterprise bounded workflow PoV candidates
```

Do not exceed 12 in the first wave.

The goal is learning and signal, not reach.

## Scoring model

Score each target from 0 to 3 on each dimension:

- Pain signal
- Timing trigger
- Authority proximity
- Category comprehension
- Warm access
- Path fit

## Send discipline

Before sending, every target must have:

- assigned path;
- trigger;
- reason Fork matters now;
- one asset anchor;
- one low-risk next step;
- explicit non-claim posture;
- status in tracker.

## Do not do

Do not:

- send the same message to all three audiences;
- lead with "AI governance framework";
- overstate compliance;
- imply production readiness;
- imply customer deployment;
- imply certification;
- send the entire repo to enterprise buyers as the first touch;
- ask for endorsement in first contact;
- send to more than 12 first-wave targets;
- commit populated target lists to the public repo.

## Success condition

The first success condition is:

Two serious conversations with people who understand the problem and can either critique, route, or scope Fork.

A serious conversation means the recipient can name a workflow, risk, governance gap, evidence problem, or buyer path that Fork should either address or explicitly avoid.
